Orchestration

Decompose complex tasks into sub-steps, delegate to specialised agents, and review results — all coordinated by CORTEX.

Overview

Orchestration is CORTEX's ability to break a high-level request into a structured multi-step plan, assign each step to the most capable sub-agent, and synthesise the results into a coherent final answer. Unlike a single-turn chat response, orchestration models the task as a directed graph of interdependent operations.

This is especially useful for tasks that require multiple skills — writing code, running shell commands, browsing the web, reading files, and then combining the output — all without you having to micromanage each step.

How It Works

When you submit a complex request, CORTEX runs through five phases:

  1. Analyse — The primary model examines your request and identifies the individual steps needed to fulfil it.
  2. Plan — A dependency graph is built so steps can run in parallel where possible.
  3. Delegate — Each step is dispatched to the most capable sub-agent (code, research, file ops, shell, etc.).
  4. Review — Results are checked against the plan. Failed steps may be retried or re-planned.
  5. Synthesise — A final summary is assembled from all sub-task results.

Orchestration runs entirely on the backend. You describe a task and CORTEX responds with the decomposed plan and final results in the chat.

When to Use Orchestration

Orchestration shines when a task requires coordination across multiple capabilities:

  • Building a full-stack feature (backend + frontend + tests)
  • Researching a topic and then generating a structured report with citations
  • Refactoring a codebase with cross-file changes
  • Setting up a project from scratch — scaffold, install dependencies, configure, verify
  • Data pipelines that involve extraction, transformation, and visualisation

For simple single-step requests (e.g., "explain this function"), orchestration adds unnecessary overhead. Use Chat mode instead.

Example

Suppose you ask CORTEX to build a full-stack microblogging app. The orchestration loop might decompose this into:

  1. Scaffold the FastAPI backend project structure
  2. Set up SQLite database with user and post models
  3. Implement authentication endpoints (register, login, JWT)
  4. Build the Vue 3 frontend project with routing
  5. Create the feed page and compose box components
  6. Write API integration layer in the frontend
  7. Run tests on the backend API
  8. Verify the full stack builds and starts

Each step is delegated to an appropriate agent. Steps 1 and 4 can run in parallel. Step 3 depends on step 2. Step 8 depends on everything else. CORTEX manages all of this automatically:

$ Build a microblogging app with a FastAPI backend, SQLite
$ database, and a Vue 3 frontend. Include user auth, a
$ feed page, and a compose box. Write tests for the API.

CORTEX returns the decomposed plan and results directly in the chat. Each completed step is marked with a green check. If a test step fails, CORTEX may retry it or ask you for guidance before continuing.

Tip: You can chain orchestration with Build mode. Start with an orchestration request like "build a landing page", and CORTEX will plan the work and hand off file creation to the Build mode agent automatically. This combines the high-level planning of orchestration with the precise diff-based editing of Build mode.
Note: Orchestration uses additional context tokens because the plan graph and sub-agent results are held in memory. For very large tasks, consider breaking the work into separate orchestration sessions. Performance depends on your underlying model — larger models (70B+) handle complex orchestration more reliably.

Tips

  • Be explicit — The more detail you give in your request, the better CORTEX can decompose the task. Include file paths, technology choices, and acceptance criteria.
  • Review the plan — CORTEX shows the proposed plan in the chat before execution starts. Review it and reorder or remove steps if needed.
  • Use Build mode for file editing — For tasks that primarily involve writing or editing source code, consider using Build mode directly. Orchestration is best when the task spans multiple domains (code + research + shell).
  • Monitor token usage — Each sub-agent call consumes tokens from your model provider. For cloud models, orchestration can add up quickly. Use local models for experimentation.
  • Pause and resume — You can pause orchestration mid-execution, inspect interim results, and resume. This is useful for approving generated code before it continues.
